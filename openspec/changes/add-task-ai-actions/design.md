## Context

PlanFlow generates AI-powered task boards from user goals. The AI pipeline currently handles goal classification, question generation, and board generation — but once a board is created, users interact with tasks through a purely manual detail panel. The backend already has a full AI chat system (LangGraph graphs, tools, pending actions, checkpointer) with no frontend consumer. This change adds:

1. **AI-generated contextual action buttons** per task (the primary request)
2. **Persistent artifacts** for AI-produced content (agreements, plans, research)
3. **Task chat UI** as the execution layer for actions

This is a cross-cutting change spanning the AI domain, board domain, and frontend.

## Goals / Non-Goals

### Goals
- Users see 2–4 contextual AI action buttons when they open any task
- Actions are generated by the LLM based on task content, status, dependencies, and board context
- Clicking an action sends a pre-filled message to the task's AI chat, which executes it
- AI-generated content is persisted as named artifacts on the task (markdown)
- Artifacts are viewable, copyable, and deletable from the task detail panel
- The task detail panel gets AI sections (actions, artifacts, chat) in a single scrollable view
- Users can refresh action suggestions to get different options

### Non-Goals
- Board-level AI actions (only task-level for now)
- Artifact editing in-place (copy-paste for MVP; in-place editing is a future feature)
- File attachments or non-text artifacts
- Collaborative artifact sharing
- Action scheduling or automation (actions are always user-triggered)
- Full chat history browsing (chat persists via checkpointer, but no explicit history pagination UI)

## Decisions

### Decision 1: Action generation as a separate lightweight endpoint
**What**: Create a dedicated `POST /api/tasks/{task_id}/actions/suggest` endpoint that returns action suggestions. This is separate from the chat endpoint.

**Why**: Action generation is a simpler, cheaper LLM call than a full chat turn. It doesn't need the LangGraph graph, tools, or checkpointer — just a structured output call with task context. Keeping it separate allows:
- Using a cheaper/faster model for suggestions (configurable)
- Caching suggestions (future optimization)
- No side effects (pure read operation)

**Alternatives considered**:
- *Generate actions via the chat system*: Would require sending a hidden "suggest actions" message, creating a chat thread just for suggestions, and parsing the response. Overly complex and expensive.
- *Hardcoded action templates*: Cheapest but defeats the purpose — actions wouldn't be contextual.

### Decision 2: Actions as pre-filled chat prompts
**What**: Each action suggestion includes a `prompt` field — a natural language message that, when the user clicks the button, gets sent to the task chat endpoint as if the user typed it.

**Why**: This reuses the entire existing chat infrastructure (LangGraph graph, tools, pending actions, checkpointer). The AI chat already knows how to update tasks, create subtasks, search the web, etc. The action button is simply a UX shortcut for typing a message.

**Example flow**:
1. Action suggestion: `{ label: "Generate rental agreement draft", prompt: "Generate a rental agreement draft based on my task details and requirements" }`
2. User clicks button → frontend sends `POST /api/tasks/{task_id}/chat` with `{ message: "<prompt>" }`
3. AI responds in chat, potentially calling the `save_artifact` tool to persist content
4. Frontend renders the chat response + artifact in the panel

### Decision 3: Artifacts as a separate DB table linked to tasks
**What**: New `Artifact` model with `id`, `task_id`, `title`, `content` (markdown text), `content_type` ("text/markdown"), `created_by` ("ai" | "user"), `created_at`. CRUD endpoints on `POST/GET/DELETE /api/tasks/{task_id}/artifacts`.

**Why**: Artifacts are distinct from chat messages — they represent finalized, named content that the user wants to keep. Storing them separately allows:
- Listing artifacts without loading chat history
- Deleting artifacts independently of chat
- Future: artifact versioning, editing, exporting

**Alternatives considered**:
- *Store in chat history only*: Users would have to scroll through chat to find content. No way to delete content without deleting chat messages. Poor UX.
- *Store in task description*: Description is a single text field; multiple artifacts don't fit cleanly.

### Decision 4: `save_artifact` as a new AI chat tool
**What**: Add a `save_artifact` tool to the task chat tool registry. When the AI generates substantial content (an agreement, a plan, a comparison), it calls this tool to persist it as an artifact.

**Why**: The AI already has a tool-calling mechanism. Adding `save_artifact` as a tool means the AI can decide when content should be saved as an artifact vs. just included in the chat response. The system prompt instructs the AI to use this tool for substantial, reusable content.

**Execution mode**: Immediate (not pending confirmation). Saving content is non-destructive.

### Decision 5: Single scrollable panel (no tabs)
**What**: The TaskDetailPanel remains a single scrollable panel with sections in this order: Status → Title → Description → Metadata → Dependencies/Unlocks → Subtasks → **AI Actions** → **Artifacts** → **Chat**.

**Why**: The user chose this layout. It keeps everything visible without tab-switching, which is important for discovery — users will see AI actions as they scroll. The chat naturally lives at the bottom, similar to a support widget.

### Decision 6: Action generation context = task + immediate dependencies
**What**: The action suggestion prompt includes the task's title, description, status, subtasks (titles + completion), and the titles/statuses of direct dependency and dependent tasks.

**Why**: This provides enough context for relevant suggestions without excessive token cost. Including the full board or goal text would increase cost significantly for marginal improvement.

## Risks / Trade-offs

- **LLM latency for action suggestions**: Mitigated by generating on task open (user expects some loading), using a fast model, and showing a skeleton loader. Future: cache suggestions per task until task changes.
- **Cost per task open**: Each task open triggers an LLM call. Mitigated by: using a cheap model (e.g., GPT-4o-mini), keeping the prompt small, and limiting to 2–4 actions. Estimated cost: <$0.001 per suggestion request.
- **AI hallucinating irrelevant actions**: Mitigated by including task context in the prompt and using structured output with a constrained schema. The worst case is a generic suggestion, which is still usable.
- **Artifact storage growth**: Artifacts are text/markdown, so storage is minimal. No file uploads or binary data in this scope.

## Migration Plan

1. Backend: Create Alembic migration for `artifact` table
2. Backend: Add artifact model, schemas, repository, service, CRUD endpoints
3. Backend: Add action suggestion endpoint + prompt
4. Backend: Add `save_artifact` tool to chat tool registry
5. Frontend: Regenerate Orval types
6. Frontend: Build AI actions component (suggestion fetching + buttons)
7. Frontend: Build artifacts section component
8. Frontend: Build chat component (message list + input + tool action cards + pending action UI)
9. Frontend: Integrate all new sections into TaskDetailPanel
10. E2E: Test full flow — open task → see actions → click action → chat executes → artifact saved → artifact visible

## Open Questions

- Should action suggestions be cached per task (invalidated on task update)? Deferred to a future optimization.
- Should we limit the number of artifacts per task? Start unlimited, add limits if storage becomes a concern.
- Should the chat input be always visible or collapsible? Start always visible at the bottom.
